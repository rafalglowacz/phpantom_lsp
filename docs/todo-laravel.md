# PHPantom — Laravel Support Plan

> This document describes the architecture and implementation plan for
> Laravel-specific intelligence in PHPantom, and the virtual member provider
> abstraction that makes it possible. The design prioritizes clean separation
> from the general PHP/PHPStan engine so that framework-specific logic never
> leaks into core resolution.

---

## Philosophy

- **No application booting.** We will never boot a Laravel application to
  resolve types. It is unreliable (database calls in boot, environment
  dependencies, Docker-only setups) and fragile.
- **No alias resolution.** Laravel service container string aliases
  (`'cache'`, `'auth'`, etc.) are not resolvable without booting. Developers
  should use `Cache::class` or `app(CacheManager::class)->` instead of
  string aliases. We will not parse application providers.
- **No SQL/migration parsing.** Model column types will not be inferred from
  database schemas or migration files. We require `@property` annotations
  on models (via ide-helper or hand-written) for column attributes.
- **No Blade (yet).** Blade template support is a large scope item that may
  come later. It is not part of this plan.
- **Larastan-style hints required.** We expect relationship methods and other
  model code to be annotated in the style that Larastan expects today. We
  target the current Laravel version's type annotations with no plans to
  support legacy versions. Poorly typed code may produce incorrect results.
  Garbage in, garbage out.
- **No Larastan stubs (yet).** We start with hardcoded knowledge of Laravel
  patterns. If we later need broader coverage we can inject Larastan stubs
  the same way we inject phpstorm-stubs. Stubs are embedded at build time
  and are effectively just as hardcoded as Rust code from the user's
  perspective. The difference is only about maintenance burden.
- **Facades fall back to `@method`.** Facades whose `getFacadeAccessor()`
  returns a string alias cannot be resolved. Facades that return `::class`
  can be resolved in the future. For now, `@method` tags on facade classes
  (generated by ide-helper or hand-written) provide completion without
  template intelligence. This is still better than every other PHP tool.

## Architecture: Virtual Member Providers

### The problem with today's design

`resolve_class_with_inheritance` in `inheritance.rs` eagerly merges three
layers:

1. Traits used by the class
2. Parent class chain
3. `@mixin` members

Additionally, `@method` and `@property` / `@property-read` / `@property-write`
tags are parsed eagerly during AST extraction and stored directly on
`ClassInfo` as if they were real declared members. This has several problems:

- **Wrong precedence.** `@method` and `@property` document `__call` / `__get`
  magic. They are not real declarations. Treating them as class-own members
  gives them higher precedence than trait and parent members, which is
  incorrect. A real method from a trait should beat a `@method` tag.
- **Eager work for nothing.** Parsing 30 `@method` tags on a Facade class
  that nobody is accessing wastes time. The docblock string is already in
  memory; we can defer parsing it until someone actually needs a member that
  is not a real declared member.
- **No override path.** A framework provider (e.g. for Laravel) may have
  richer type information than a `@method` tag. For example, ide-helper
  generates `@method static string get(string $key)` on a Facade, but the
  underlying service class has `@template TValue @return TValue|null`. If
  `@method` sits at class-own precedence, the framework provider can never
  win.
- **Ambiguous ordering.** There is no spec that defines precedence between
  `@method`, `@mixin`, and framework magic. All of them describe virtual
  members provided via PHP's magic methods. The only hard line is: real
  declared members beat everything else. After that, it is a free-for-all
  that depends on debugging and reading code. Putting all virtual sources
  behind one abstraction with an explicit priority order makes the behavior
  predictable.

### The abstraction

All three sources of virtual members (`@method`/`@property`, `@mixin`, and
framework magic) are unified behind a **virtual member provider** pattern.

The resolver performs base resolution (own members + traits + parent chain)
to produce a "base resolved" class, then queries registered providers lazily
(only when completion or go-to-definition actually needs members beyond the
base class). Providers are queried in priority order; earlier providers'
members shadow later ones.

```rust
/// Members synthesized by a provider. Merged below real declared members,
/// traits, and parent chain.
pub struct VirtualMembers {
    pub methods: Vec<MethodInfo>,
    pub properties: Vec<PropertyInfo>,
}

/// A provider that contributes virtual members to a class.
///
/// Receives the class with traits and parents already merged,
/// but WITHOUT other providers' contributions (no circular loading).
pub trait VirtualMemberProvider {
    /// Whether this provider has anything to say about this class.
    /// Cheap check so the resolver can skip providers early.
    fn applies_to(
        &self,
        class: &ClassInfo,
        class_loader: &dyn Fn(&str) -> Option<ClassInfo>,
    ) -> bool;

    /// Produce virtual members for this class.
    fn provide(
        &self,
        class: &ClassInfo,
        class_loader: &dyn Fn(&str) -> Option<ClassInfo>,
    ) -> VirtualMembers;
}
```

### Precedence model

```
1. Real declared members (in PHP source code)
2. Trait members (real implementations)
3. Parent chain members (real implementations)
4. Virtual member providers (in priority order):
   a. Framework provider  — richest type info (Laravel relationships, scopes, builder)
   b. PHPDoc provider     — @method, @property, @property-read, @property-write
   c. Mixin provider      — @mixin
```

Within step 4, providers are queried in order. A member contributed by a
higher-priority provider is not overwritten by a lower-priority one.

This means:
- A Laravel relationship property synthesized from `hasMany(Post::class)`
  beats a `@property` tag from ide-helper for the same name (the
  synthesized version carries generic type information that the tag cannot
  express).
- A `@method` tag beats a same-named method from a `@mixin` class. Note
  that `@mixin` is actually the richer source (it pulls in the real method
  with full generics). If someone wrote `@method` on top of `@mixin`, they
  are presumably working around a conflict in the mixin, or catering to a
  tool that does not understand `@mixin`. Either way, respecting the
  explicit annotation is the safer default.
- All virtual members lose to real declared members, trait members, and
  parent chain members.

### Resolution flow

```
1. resolve_class_with_inheritance(User)
   → own members + traits + parent chain
   → returns "base resolved" ClassInfo
   → @method/@property tags are NOT parsed here, the raw docblock
     string is preserved on ClassInfo for later

2. For each registered provider (in priority order):
   if provider.applies_to(base_class):
       virtual = provider.provide(base_class)
       merge virtual into base_class (no overwrites of existing members)
```

Step 2 is lazy. It only runs when someone triggers completion or
go-to-definition, not during parsing or indexing. This means:
- Scanning a model to learn about its scopes and relationships does not
  trigger loading the entire Builder tree via `@mixin`.
- The 30 `@method` tags on a Facade class are not parsed until someone
  actually completes on that Facade.

### File layout

```
src/
├── virtual_members/
│   ├── mod.rs           # VirtualMemberProvider trait, VirtualMembers struct,
│   │                    # merge logic, provider registry
│   ├── phpdoc.rs        # PHPDocProvider (@method, @property, @property-read, @property-write)
│   ├── mixin.rs         # MixinProvider (extracted from inheritance.rs)
│   └── laravel.rs       # LaravelModelProvider
├── inheritance.rs       # Base resolution only (traits + parent chain)
```

The general code in `resolver.rs`, `handler.rs`, `builder.rs`, and
`definition/` never imports `laravel.rs` or `phpdoc.rs`. It only interacts
with the `VirtualMemberProvider` trait.

## Provider 1: PHPDocProvider

Extracted from the current eager parsing in `parser/classes.rs` where
`extract_property_tags` and `extract_method_tags` are called during AST
extraction and their results are stuffed directly into `ClassInfo.properties`
and `ClassInfo.methods`.

### What changes

- During AST extraction, the raw class-level docblock string is preserved on
  `ClassInfo` (it may already be, or we store the relevant slice). The
  `@method` and `@property` tags are **not** parsed eagerly into
  `ClassInfo.methods` / `ClassInfo.properties`.
- Real declared properties and methods (those with actual PHP declarations
  in the class body) remain on `ClassInfo` as before.
- The PHPDocProvider parses `@method` / `@property` / `@property-read` /
  `@property-write` tags lazily when `provide` is called.

### Provider behavior

- **`applies_to`**: class has a non-empty class-level docblock. (Cheap
  string-presence check, no parsing.)
- **`provide`**: parse the docblock for `@method` and `@property` tags using
  the existing `extract_method_tags` and `extract_property_tags` functions.
  Return the results as `VirtualMembers`.

### Why this is better

- Docblock parsing is deferred until needed.
- `@method` / `@property` members sit at the correct precedence (below real
  members, traits, and parents; above `@mixin`).
- Framework providers can override them with richer type information.
- The parsing functions (`extract_method_tags`, `extract_property_tags`)
  remain unchanged. Only the call site moves.

---

## Provider 2: MixinProvider

Extracted from the current `merge_mixins_into` / `merge_mixins_into_recursive`
logic in `inheritance.rs`. Behavior is identical to today, just shaped as a
provider and sitting at the lowest priority.

- **`applies_to`**: class or any ancestor has a non-empty `mixins` vec.
- **`provide`**: loads each mixin class, resolves its inheritance (traits +
  parents, not other providers), returns public members. Recurses into
  mixins declared on mixin classes up to the existing depth limit.

This is a pure refactor. No new behavior, no new tests beyond confirming
existing tests still pass.

## Provider 3: LaravelModelProvider

### Detection

**`applies_to`**: walk the `parent_class` chain via `class_loader` looking
for `Illuminate\Database\Eloquent\Model`. Cache the result per class name
per session to avoid repeated walks.

### What it synthesizes

The provider scans the base-resolved class (own methods + traits + parent
chain, no virtual members) and produces three categories of virtual members:

#### A. Relationship properties

Scan methods whose return type is a known Eloquent relationship class.
For each such method, synthesize a virtual property.

**Known relationship classes and their property types:**

| Relationship class                  | Virtual property type          |
|-------------------------------------|--------------------------------|
| `HasOne`, `MorphOne`, `BelongsTo`   | `TRelated` (singular, nullable)|
| `HasMany`, `MorphMany`, `BelongsToMany`, `HasManyThrough`, `MorphToMany` | `\Illuminate\Database\Eloquent\Collection<TRelated>` |
| `MorphTo`                           | `\Illuminate\Database\Eloquent\Model` |

**How `TRelated` is determined:**

Read the method's docblock return type. We expect Larastan-style annotations:

```php
/**
 * @return HasOne<UserAgent, $this>
 */
public function userAgent(): HasOne
```

The first generic parameter of the relationship type is `TRelated`. Extract
it with the existing `split_generic_args` utility.

If there is no generic parameter on the return type, fall back to checking
the method body text for the first `::class` argument to the relationship
call (e.g. `$this->hasMany(Post::class)` → `Post`). This is a best-effort
heuristic for codebases that haven't adopted Larastan-style annotations.

**Virtual property name:** the method name itself. `userAgent()` →
`$userAgent`.

#### B. Scope methods

Scan methods whose name starts with `scope` (case-sensitive, e.g.
`scopeActive`, `scopeVerified`).

For each scope method:
- Strip the `scope` prefix and lowercase the first letter: `scopeActive` →
  `active`.
- Create a virtual method with that name.
- The method is available as **both static and instance** (Laravel allows
  `User::active()` and `$query->active()`).
- Strip the first parameter (`$query`) from the parameter list since Laravel
  injects it.
- Return type: use the scope method's declared return type. If it returns
  `void` or has no return type, default to
  `\Illuminate\Database\Eloquent\Builder<static>`.

#### C. Builder-as-static forwarding

The real `Illuminate\Database\Eloquent\Model` does not have `@mixin Builder`.
Instead, `Model::query()` returns `\Illuminate\Database\Eloquent\Builder<static>`,
and `Model::__callStatic()` forwards static calls to `static::query()->`.

The provider synthesizes this forwarding:
1. Load `\Illuminate\Database\Eloquent\Builder`.
2. Resolve its inheritance (own + traits + parents).
3. Take its public instance methods.
4. Present them as **static** virtual methods on the model class.
5. Return types that reference `static` or `$this` on Builder should resolve
   to the Builder itself (since the chain continues on the builder, not the
   model). Return types that reference the model template parameter should
   resolve to the concrete model class.

This makes `User::where('email', $e)->orderBy('name')->get()` resolve
end-to-end.

## Implementation sequence

### Step 5: LaravelModelProvider — Builder-as-static

- Load Eloquent Builder, extract public instance methods.
- Present as static virtual methods on Model subclasses.
- Handle return type mapping (`static` on Builder → Builder continuation,
  model template parameter → concrete model class).
- Tests: verify `User::where()->orderBy()->get()` chain resolves.

### Step 6: Completion builder adjustments

- Virtual methods that are marked as "static forwarded" need to appear in
  `DoubleColon` completion results even though the underlying method is an
  instance method. The `build_completion_items` filter in `builder.rs`
  should respect whatever flag the provider sets.
- Virtual properties from relationships should appear in `Arrow` completion.

### Step 7: Go-to-definition support

- When go-to-definition lands on a virtual property (relationship), jump to
  the relationship method definition.
- When go-to-definition lands on a virtual method (scope), jump to the
  `scope*` method definition.
- When go-to-definition lands on a forwarded Builder method, jump to the
  Builder method definition.

## Out of scope (and why)

| Item | Reason |
|------|--------|
| Container string aliases | Requires booting the application. Use `::class` references instead. |
| Facade `getFacadeAccessor()` with string aliases | Same problem. `@method` tags provide a workable fallback. |
| Blade templates | Large scope, separate project. May come later. |
| Model column types from DB/migrations | Unreasonable complexity. Require `@property` annotations. |
| Legacy Laravel versions | We target current Larastan-style annotations. Older code may degrade. |
| Application provider scanning | Low-value, high-complexity. Maybe someday if someone twists our arm. |
| `ide-helper` integration | It generates annotations that conflict with proper tooling. We read its `@property` tags like any other docblock, but do not depend on or recommend its other output. |

## Testing strategy

Each step gets its own test coverage:

- **Provider abstraction**: unit tests for merge logic (priority ordering,
  no-overwrite semantics).
- **MixinProvider refactor**: all existing `completion_inheritance` and
  `completion_mixin` tests pass unchanged.
- **PHPDocProvider refactor**: all existing `@method` / `@property` tests
  pass unchanged. Verify that `@method` members no longer appear on
  `ClassInfo.methods` before provider resolution. Verify that a real
  declared method beats a same-named `@method` tag. Verify that a trait
  method beats a same-named `@method` tag.
- **Provider priority**: a LaravelModelProvider member beats a PHPDocProvider
  member for the same name. A PHPDocProvider member beats a MixinProvider
  member for the same name.
- **Relationship properties**: mock Eloquent Model + relationship classes,
  verify property completion on model instances, verify generic type
  extraction from Larastan-style annotations, verify fallback to `::class`
  argument parsing.
- **Scope methods**: mock model with `scopeActive`, `scopeVerified`, verify
  completion for both `User::active()` and `$query->active()`, verify
  `$query` parameter is stripped.
- **Builder forwarding**: mock Eloquent Builder with `where`, `orderBy`,
  `get`, verify `User::where()->` chain resolves, verify return types map
  correctly.
- **Cross-file PSR-4**: all of the above with model and relationship classes
  in separate files.
- **Trait scopes/relationships**: scopes and relationships defined in traits
  used by the model are discovered.
